# 成本与可用性分析

## 成本估算

### 云端 LLM 推理成本

**使用国内厂商 API（推荐方案）**

#### 主流厂商定价对比（2026年3月）

| 厂商 | 模型 | 输入价格 | 输出价格 | 特点 |
|------|------|---------|---------|------|
| 阿里云 | 通义千问-Plus | ¥0.004/1K tokens | ¥0.012/1K tokens | 性能强，适合复杂对话 |
| 阿里云 | 通义千问-Turbo | ¥0.0008/1K tokens | ¥0.002/1K tokens | 性价比高，适合简单对话 |
| 百度 | 文心一言 4.0 | ¥0.012/1K tokens | ¥0.012/1K tokens | 中文理解好 |
| 字节 | 豆包-Pro | ¥0.0008/1K tokens | ¥0.002/1K tokens | 低延迟 |
| 智谱 | GLM-4 | ¥0.01/1K tokens | ¥0.01/1K tokens | 推理能力强 |

#### 成本估算（以通义千问-Turbo 为例）

**单次对话成本**：
- 输入：500 tokens × ¥0.0008 = ¥0.0004
- 输出：200 tokens × ¥0.002 = ¥0.0004
- 合计：¥0.0008/次

**月度成本**：
- 每天 50 次对话：¥0.04/天
- 月成本：¥1.2/月/用户

**不同用户规模**：
- 10 用户：¥12/月
- 100 用户：¥120/月
- 1000 用户：¥1,200/月

#### 优势
1. **零运维成本**：无需管理 GPU 服务器
2. **弹性扩展**：按需付费，自动扩容
3. **低延迟**：国内厂商节点多，网络快
4. **持续优化**：厂商持续优化模型性能

**推荐策略**：
- 全阶段使用 API（除非月成本 >¥10,000）
- 简单对话用 Turbo，复杂任务用 Plus
- 智能路由节省 30-50% 成本

## 可用性设计

### 1. 网络依赖问题

**风险**：断网后完全无法使用

**降级方案**：

#### 方案 1：本地规则引擎（推荐）
```
设备缓存常见对话模板：
- "我想玩过家家" → "好呀，我们玩什么呢？"
- "讲个故事" → 播放预存的故事音频
- "我爱你" → "我也爱你呀"

优点：零延迟，离线可用
缺点：缺少个性化和记忆
```

#### 方案 2：轻量级本地模型
```
嵌入式设备运行小模型（<1B）：
- Qwen2.5-0.5B / TinyLlama
- 仅基础对话能力
- 网络恢复后同步 Memory

优点：有一定智能
缺点：需要更强硬件（树莓派 4B+）
```

#### 方案 3：离线提示
```
"网络断开了，等网络好了我们再聊吧"
播放预存的儿歌/故事

优点：实现简单
缺点：用户体验差
```

**推荐**：方案 1（阶段一、二）+ 方案 2（阶段三可选）

### 2. 服务可用性保障

#### 架构冗余
```
┌─────────┐
│ 设备    │
└────┬────┘
     │
     ▼
┌─────────────────┐
│ 负载均衡 (SLB)  │
└────┬────────────┘
     │
     ├──► HTTP 代理 1 ──► Gateway 1 ──► LLM 服务 1
     ├──► HTTP 代理 2 ──► Gateway 2 ──► LLM 服务 2
     └──► HTTP 代理 3 ──► Gateway 3 ──► LLM 服务 3
```

#### 故障处理
- **代理故障**：负载均衡自动切换
- **Gateway 故障**：代理重试其他实例
- **LLM 超时**：降级到缓存响应
- **Memory 故障**：临时内存模式

#### 监控指标
- 可用性目标：99.5%（允许每月 3.6 小时故障）
- 延迟 P95：<3s
- 错误率：<1%

## 成本优化策略

### 1. 缓存策略
```
常见问题缓存（Redis）：
- "你叫什么名字？"
- "我们玩什么？"
- "讲个故事"

命中率 20% → 节省 20% LLM 成本
```

### 2. 模型选择
```
简单对话：qwen2.5-7b（成本 1/4）
复杂任务：qwen2.5-32b
教学场景：qwen2.5-14b（平衡）

智能路由 → 节省 40-60% 成本
```

### 3. 批处理优化
```
多用户请求批量推理（batch inference）
吞吐量提升 2-3x
```
