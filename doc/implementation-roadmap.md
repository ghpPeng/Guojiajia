#### 第 4-5 个月：硬件选型和开发

**硬件选型**:
```
方案 A: 树莓派 Zero 2W
  - CPU: 4核 1GHz
  - RAM: 512MB
  - 价格: ¥80
  - 优点: 生态成熟，开发简单
  - 缺点: 功耗较高

方案 B: ESP32-S3
  - CPU: 双核 240MHz
  - RAM: 512KB
  - 价格: ¥30
  - 优点: 低功耗，成本低
  - 缺点: 性能有限

推荐: 方案 A（阶段二），方案 B（阶段三优化）
```

## 六、开发规范

### 6.1 代码规范

```yaml
代码风格:
  - JavaScript/TypeScript: Airbnb Style Guide
  - Kotlin: Official Kotlin Style Guide
  - 使用 ESLint/Prettier/ktlint 自动格式化

Git 工作流:
  - 主分支: main
  - 开发分支: develop
  - 功能分支: feature/xxx
  - 修复分支: fix/xxx
  - Commit 格式: Conventional Commits

代码审查:
  - 所有代码必须经过 Code Review
  - 至少一个 Approve 才能合并
  - CI 必须通过（测试 + 构建）
```

### 6.2 测试要求

```yaml
测试覆盖率:
  - 单元测试: >80%
  - 集成测试: 核心流程 100%
  - E2E 测试: 关键用户路径

测试工具:
  - 后端: Jest + Supertest
  - Android: JUnit + Espresso
  - E2E: Playwright（Web）/ Appium（Android）
```

### 6.3 文档要求

```yaml
必需文档:
  - API 文档（Swagger/OpenAPI）
  - 部署文档
  - 运维手册
  - 用户手册
  - 开发者指南

文档更新:
  - 代码变更必须同步更新文档
  - 每个 Sprint 结束前更新文档
```

## 七、风险管理

### 7.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| LLM API 不稳定 | 高 | 中 | 多厂商备份，降级策略 |
| Memory 性能瓶颈 | 中 | 中 | 提前压测，优化查询 |
| 多设备同步冲突 | 中 | 高 | CRDT 算法，充分测试 |
| 硬件成本过高 | 高 | 低 | 多方案对比，批量采购 |

### 7.2 业务风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户增长缓慢 | 高 | 中 | 加强推广，优化体验 |
| 内容安全问题 | 高 | 中 | 严格过滤，人工审核 |
| 隐私合规问题 | 高 | 低 | 法律咨询，合规设计 |
| 竞品压力 | 中 | 高 | 差异化定位，快速迭代 |
