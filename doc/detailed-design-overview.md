## 二、设计亮点

### 2.1 解决的核心问题

本次详细设计解决了架构审查中发现的 **P0 级别问题**：

#### ✅ 问题 1: Memory 架构重新设计

**原问题**: Redis 单机有容量上限，MongoDB 分片策略未说明，多设备同步未设计

**解决方案**:
- 三层架构：L1 (Redis) → L2 (MongoDB) → L3 (OSS)
- 一致性哈希分片，支持平滑扩容
- 自动数据生命周期管理
- 完整的故障降级方案

**文档**: [detailed-design-memory.md](./detailed-design-memory.md)

---

#### ✅ 问题 2: 认证授权方案加强

**原问题**: Token 生成/刷新机制不明确，Token 泄露无应对方案，缺少设备指纹

**解决方案**:
- OAuth2 + JWT 标准流程
- access_token (1h) + refresh_token (30d)
- 设备指纹验证，防止设备替换
- Token 撤销和黑名单机制
- 家长扫码绑定流程

**文档**: [detailed-design-auth.md](./detailed-design-auth.md)

---

#### ✅ 问题 3: 多设备同步提前设计

**原问题**: 阶段三才考虑，可能推倒重来，冲突解决策略缺失

**解决方案**:
- 混合同步策略（LWW + CRDT）
- 字段级冲突解决
- 版本向量检测冲突
- 离线同步支持
- 完整的测试用例

**文档**: [detailed-design-sync.md](./detailed-design-sync.md)

---

### 2.2 解决的重要问题 (P1)

#### ✅ 问题 4: HTTP 代理与 Gateway 职责明确

**解决方案**:
- 代理层：只做安全（认证/过滤/限流/监控）
- Gateway：只做业务（Agent 路由/Memory/LLM）
- 清晰的接口定义和数据流

**文档**: [detailed-design-proxy.md](./detailed-design-proxy.md)

---

#### ✅ 问题 5: 失败场景补充设计

**解决方案**:
- LLM API 超时重试 + 指数退避
- 限流队列和熔断器
- Memory 故障降级到本地缓存
- 网络故障离线模式
- 完整的错误码处理

**文档**: [detailed-design-failure-handling.md](./detailed-design-failure-handling.md)

---

#### ✅ 问题 6: 监控告警提前规划

**解决方案**:
- 五大类指标（业务/性能/资源/错误/成本）
- Prometheus + Grafana + Loki 技术栈
- 分级告警规则（P0/P1/P2）
- 多渠道通知（钉钉/邮件/短信）
- 完整的仪表盘设计

**文档**: [detailed-design-monitoring.md](./detailed-design-monitoring.md)

### 3.2 数据流图

```
用户说话
   ↓
设备本地 STT → 文本
   ↓
HTTPS POST 到代理
   {device_id, token, text}
   ↓
代理层处理:
   1. 验证 JWT Token
   2. 验证设备指纹
   3. Rate Limiting 检查
   4. 内容安全过滤
   5. 记录访问日志
   ↓
转发到 Gateway
   ↓
Gateway 处理:
   1. 从 Memory 读取上下文
   2. 路由到 child-doll Agent
   3. 调用 LLM API
   4. 更新 Memory
   ↓
LLM Streaming 响应
   ↓
代理层转发
   {text, emotion, chunk_id}
   ↓
设备本地 TTS → 语音
   ↓
播放 + 更新表情
```

## 四、关键技术决策

### 4.1 Memory 系统

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 分层策略 | 三层（Redis/MongoDB/OSS） | 平衡性能和成本 |
| 分片算法 | 一致性哈希 | 支持平滑扩容 |
| 数据迁移 | 自动 TTL + 定时任务 | 降低运维成本 |
| 故障降级 | 内存 Map | 保证服务可用性 |

### 4.2 认证授权

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 认证协议 | OAuth2 + JWT | 行业标准，安全可靠 |
| Token 有效期 | access 1h, refresh 30d | 平衡安全和体验 |
| 设备验证 | 设备指纹 | 防止设备替换攻击 |
| 家长绑定 | 扫码 + 验证码 | 简单易用 |

### 4.3 多设备同步

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 同步策略 | 混合（LWW + CRDT） | 平衡复杂度和正确性 |
| 冲突检测 | 版本向量 | 准确检测并发修改 |
| 冲突解决 | 字段级合并 | 最大化保留数据 |
| 离线支持 | 本地队列 + 重放 | 支持弱网环境 |

### 4.4 HTTP 代理

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 技术栈 | Node.js + Express | 开发效率高，生态丰富 |
| 通信协议 | HTTPS + WebSocket | 安全 + 实时 |
| 限流算法 | 令牌桶 | 平滑限流，支持突发 |
| 降级策略 | 缓存 + 规则 | 保证基础可用性 |

### 4.5 监控告警

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 监控系统 | Prometheus + Grafana | 开源，功能强大 |
| 日志系统 | Loki | 与 Grafana 集成好 |
| 告警通道 | 钉钉 + 邮件 + 短信 | 多渠道保证送达 |
| 指标保留 | 30-365天 | 平衡存储成本 |

## 六、风险与应对

### 6.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| LLM API 不稳定 | 高 | 中 | 多厂商备份，降级策略 |
| Memory 性能瓶颈 | 中 | 低 | 分片扩容，缓存优化 |

| 网络延迟过高 | 高 | 低 | CDN 加速，边缘计算 |

### 6.2 业务风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户增长过快 | 高 | 低 | 弹性扩容，限流 |
| 成本超预算 | 中 | 中 | 成本监控，智能路由 |
| 内容安全问题 | 高 | 低 | 多层过滤，人工审核 |
| 隐私合规问题 | 高 | 低 | 数据加密，家长授权 |
