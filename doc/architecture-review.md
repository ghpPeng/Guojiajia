## 三、核心问题（P0 - 必须解决）

### 1. Memory 架构需重新设计 🔴

**当前问题**:
- Redis 单机有容量上限
- MongoDB 分片策略未说明
- 多设备同步未设计

**改进方案**:
```
L1: Redis（热数据，<1h）
L2: MongoDB（温数据，<7d）
L3: OSS（冷数据，>7d）

分片策略: 按 device_id hash 分片
同步方案: CRDT 或 Last-Write-Wins
```

**优先级**: P0（阶段一必须解决）

---

### 2. 认证授权方案需加强 🔴

**当前问题**:
- Token 生成/刷新机制不明确
- Token 泄露无应对方案
- 缺少设备指纹

**改进方案**:
```
使用 OAuth2 + JWT:
- access_token: 短期（1h）
- refresh_token: 长期（30d）
- device_secret: 设备唯一密钥
- device_fingerprint: 设备指纹
```

**优先级**: P0（阶段一必须解决）

---

### 3. 多设备同步需提前设计 🔴

**当前问题**:
- 阶段三才考虑，可能推倒重来
- 手机 + 玩偶状态同步未设计
- 冲突解决策略缺失

**改进方案**:
```
使用 CRDT（Conflict-free Replicated Data Types）:
- 每个操作带时间戳
- 自动合并冲突
- 最终一致性

或使用 Last-Write-Wins:
- 按时间戳决定最新状态
- 简单但可能丢失数据
```

**优先级**: P0（阶段一设计，阶段三实现）

## 四、重要问题（P1 - 应该解决）

### 4. HTTP 代理与 Gateway 职责重叠 🟡

**问题**: 两层都做路由，增加复杂度和延迟

**建议**: 
- 代理层：只做安全（鉴权、过滤、限流）
- Gateway：只做业务（Agent 路由、工具调用）

---

### 5. 失败场景需补充设计 🟡

**缺失场景**:
- LLM API 超时/限流
- Memory 不一致
- 多设备并发冲突

**建议**: 补充降级、熔断、重试策略

---

### 6. 监控告警需提前规划 🟡

**建议**: 
- 接入 Prometheus + Grafana
- 监控延迟、错误率、成本
- 设置告警阈值
